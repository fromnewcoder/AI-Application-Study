The pipeline is identical regardless of index type — only the internal retrieval strategy differs. Swapping index type is a one-line change.

RetrieverQueryEngine

Standard retrieval

Retrieves top-k nodes by similarity, feeds them to an LLM for synthesis. The default. Use for most Q&A.

engine = index.as_query_engine( similarity_top_k=5, response_mode="compact" )
CitationQueryEngine

With source citations

Wraps each retrieved node in a citation. The LLM is prompted to reference sources by number — great for auditable answers.

from llama_index.core.query_engine import ( CitationQueryEngine) engine = CitationQueryEngine.from_args( index, similarity_top_k=5)
SubQuestionQueryEngine

Decompose complex queries

Breaks a hard question into sub-questions, runs each against a separate index, then synthesises. Ideal for multi-document comparison.

TransformQueryEngine

Query transformation

Wraps any engine with a pre-processor (HyDE, step-back prompting, query rewriting) before hitting the underlying retriever.

Response modes
compact

Stuffs as many nodes as fit into one LLM call. Fast, cheap.

refine

Iterates through nodes, refining the answer. Higher quality, more calls.

tree_summarize

Recursively merges node summaries. Best for large context / summaries.

A router sits in front of multiple query engines or retrievers and picks which one(s) to call based on the query. This is LlamaIndex's answer to multi-index RAG.

RouterQueryEngine

Single best match

Selects exactly one query engine from a list. The LLM reads tool descriptions and picks the most relevant. Equivalent to an intent classifier backed by the LLM.

+ Fast, one retrieval path
− Wrong if query spans multiple indexes
SelectorQueryEngine

Multi-select

Can select multiple engines and merge results. Use when a query legitimately draws from more than one data source (e.g. "compare our Q3 report vs analyst notes").

+ Handles cross-index queries
− More LLM calls, harder to synthesise
Router setup pattern
from llama_index.core.query_engine import RouterQueryEngine from llama_index.core.selectors import LLMSingleSelector from llama_index.core.tools import QueryEngineTool tools = [ QueryEngineTool.from_defaults( query_engine=vector_engine, description="Useful for specific factual questions" ), QueryEngineTool.from_defaults( query_engine=summary_engine, description="Useful for summarising large documents" ), ] router = RouterQueryEngine( selector=LLMSingleSelector.from_defaults(), query_engine_tools=tools, )

Both frameworks build RAG pipelines. The differences are architectural philosophy, not capability. Green border = better fit for that criterion.

LlamaIndex
LangChain

RAG-first design — every primitive (index, retriever, query engine) is purpose-built for document Q&A. Less setup friction for RAG use cases.

RAG

RAG is one of many use cases. More boilerplate to assemble the same pipeline — but extremely flexible.

Index types are first-class citizens: swap VectorStoreIndex for KnowledgeGraphIndex in one line. Easy multi-index routing.

Indexes

More vector store integrations (Pinecone, Weaviate, Chroma, Qdrant, Milvus…). Better for teams already invested in a specific store.

Query engines are composable — SubQuestion, Citation, Transform all wrap each other cleanly. Router pattern is built-in.

Query logic

Chains and agents are more powerful but require more explicit wiring. LCEL (LangChain Expression Language) is expressive but has a learning curve.

Agents exist (ReActAgent) but tool ecosystem is narrower. Best for retrieval-centric agents.

Agents

Richer agent ecosystem — LangGraph for stateful agents, broader tool integrations (SQL, APIs, Python REPL, web search).

LlamaTrace / Arize for observability. Logging is straightforward.

Observability

LangSmith is best-in-class for tracing, evaluation, and prompt management. Significant advantage in production.

Cleaner abstractions for RAG — fewer concepts to learn before being productive. Node parsers, postprocessors, and response synthesisers map directly to RAG stages.

DX

Steeper initial curve. Pays off with more control over complex pipelines and agent behaviour.

Decision rule

Build a document Q&A product, knowledge base, or retrieval-heavy app → LlamaIndex. Build a complex agent, multi-step workflow, or production system needing deep observability → LangChain. Many teams use both: LlamaIndex for retrieval, LangChain for orchestration.

Minimal VectorStoreIndex RAG
from llama_index.core import VectorStoreIndex, SimpleDirectoryReader from llama_index.llms.anthropic import Anthropic llm = Anthropic(model="claude-sonnet-4-20250514") docs = SimpleDirectoryReader("./data").load_data() index = VectorStoreIndex.from_documents(docs) engine = index.as_query_engine(llm=llm, similarity_top_k=5) response = engine.query("What is the refund policy?") print(response)
SummaryIndex for long-doc summarisation
from llama_index.core import SummaryIndex summary_index = SummaryIndex.from_documents(docs) summary_engine = summary_index.as_query_engine( response_mode="tree_summarize", use_async=True, ) response = await summary_engine.aquery("Summarise the entire document")
Postprocessor pipeline (reranking + similarity threshold)
from llama_index.core.postprocessor import ( SimilarityPostprocessor, SentenceTransformerRerank, ) reranker = SentenceTransformerRerank( model="cross-encoder/ms-marco-MiniLM-L-6-v2", top_n=5, ) sim_filter = SimilarityPostprocessor(similarity_cutoff=0.7) engine = index.as_query_engine( similarity_top_k=20, node_postprocessors=[sim_filter, reranker], )
Hybrid BM25 + vector with custom retriever
from llama_index.retrievers.bm25 import BM25Retriever from llama_index.core.retrievers import QueryFusionRetriever bm25 = BM25Retriever.from_defaults(index=index, similarity_top_k=10) vector = index.as_retriever(similarity_top_k=10) hybrid = QueryFusionRetriever( retrievers=[bm25, vector], similarity_top_k=5, mode="reciprocal_rerank", # RRF fusion built in use_async=True, ) from llama_index.core.query_engine import RetrieverQueryEngine engine = RetrieverQueryEngine.from_args(hybrid, llm=llm)
