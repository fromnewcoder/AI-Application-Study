What is an eval harness?

An eval harness is the scaffolding that runs your test suite automatically: it feeds inputs to your LLM pipeline, collects outputs, scores them with an LLM judge or deterministic metric, detects regressions against a prior run, and produces a report — all without manual effort.

Component
Test dataset
Curated Q&A pairs with ground-truth answers and metadata (category, difficulty, expected behavior).
Component
Runner
Iterates over test cases, calls your pipeline, and captures raw outputs + latency.
Component
Scorer
Applies metrics: exact match, F1, or an LLM-as-judge prompt returning a structured score.
Component
Regression detector
Compares this run's scores to a baseline snapshot. Flags any metric that drops below a threshold.
Component
Reporter
Aggregates results into a summary: pass/fail per case, per-category breakdowns, trend over time.
Component
CI gate
Blocks deployment if overall score drops or regression threshold is breached. Evals as code.

The eval loop

1
Build golden test set
Diverse human-labeled Q&A pairs covering happy path, edge cases, and adversarial inputs.
2
Run pipeline on every case
Feed each question through your full RAG system. Record the generated answer and latency.
3
Score automatically
LLM judge evaluates faithfulness, relevance, completeness. Deterministic metrics handle exact match.
4
Detect regressions
Compare to baseline. Any metric declining more than 5% is a regression — investigate before shipping.
5
Report and iterate
Publish scores. Fix failing cases. Re-run. The loop runs on every pull request.

LLM-as-judge prompt pattern

The judge prompt is the heart of automated scoring. It must be specific, rubric-based, and return structured output — not free text. Always ask for a score AND a reason. The reason lets you debug failing cases.

SYSTEM: You are an expert evaluator for a RAG system. Score the answer strictly on the rubric below. Return JSON only — no preamble. USER: Question: {question} Retrieved context: {context} Model answer: {model_answer} Reference answer: {reference_answer} Score each dimension 1–5: - faithfulness: every claim is grounded in context (5=fully grounded) - relevance: answer directly addresses the question (5=fully relevant) - completeness: all key aspects of reference answer are covered (5=complete) Respond ONLY with: {"faithfulness": N, "relevance": N, "completeness": N, "reasoning": "one sentence per dimension"}

Scoring modes

Mode
Absolute scoring
Judge rates the answer on a 1–5 scale against the rubric and reference answer. Best for regression detection — scores are stable across runs.
Use by default
Mode
Pairwise comparison
Judge chooses which of two answers (A vs B) is better. Useful for A/B testing prompt versions or models. Less stable for regression.
Use for A/B
Mode
Binary pass/fail
Judge answers a yes/no question: "Is this answer faithful?" Fastest and cheapest. Best for unanswerable-question detection.
Use for safety checks

Harness pseudocode

results = [] for case in test_set: answer, latency = pipeline.run(case.question) scores = llm_judge(case.question, answer, case.reference_answer, context) results.append({**case, "answer": answer, "latency": latency, **scores}) summary = aggregate(results) regressions = detect_regressions(summary, baseline) report(summary, regressions) if regressions: raise CIFailure("Regression detected")

Regression detection

A regression is when a metric drops meaningfully between two eval runs — usually across a code change, model update, or prompt edit. A harness catches these automatically so they never reach production.

Strategy
Baseline snapshot
Save your last "good" eval run as a JSON file in git. The harness diffs new results against it.
Strategy
Threshold gate
Define a minimum acceptable score per metric. Any run that dips below fails the CI check.
Strategy
Delta alert
Flag any metric that drops more than N% from baseline — even if still above the absolute threshold.
Strategy
Per-category tracking
Regressions often hide in subcategories. Track scores by category (e.g. unanswerable, multi-hop) separately.

Reporting best practices

PracticeRationale
Store all results as versioned JSONEnables long-term trend analysis and audit trails.
Report per-category breakdownsGlobal averages mask category-specific regressions.
Show raw score + delta + statusOne number without context invites misinterpretation.
Attach example failuresLink the 3 worst-scoring cases for immediate debugging.
Run on every PR, not just releasesRegressions are cheapest to fix closest to the change.
