Tracing setup

Three environment variables + zero code changes for LangChain projects.

Step 1 Set environment variables ▾
# .env or shell export export LANGCHAIN_TRACING_V2="true" export LANGCHAIN_API_KEY="ls__..." export LANGCHAIN_PROJECT="my-agent"
LANGCHAIN_PROJECT controls the LangSmith project bucket. Omitting it sends traces to default. Use a distinct name per environment (dev / staging / prod).
When it works
Any LangChain (v0.1+) call will auto-trace once these three vars are set.
When it doesn't
Async code in some frameworks needs LANGCHAIN_CALLBACKS_BACKGROUND=true to prevent early exit before flush.
Step 2 Manual tracing with @traceable ▾
from langsmith import traceable # Decorating any Python function creates a named run @traceable(run_type="chain", name="retrieve_and_rank") def retrieve_and_rank(query: str) -> list: results = vector_store.similarity_search(query) return rerank(results)
Use run_type to classify spans: chain, retriever, tool, llm, or embedding. This controls the icon shown in the LangSmith UI.
Step 3 Full agent tracing pattern ▾
from langsmith import traceable, get_current_run_tree from langchain.agents import AgentExecutor @traceable(run_type="chain", name="agent_root") def run_agent(user_input: str) -> str: # All nested LangChain calls are auto-attributed as children result = agent_executor.invoke({"input": user_input}) return result["output"]
Wrapping your top-level function with @traceable establishes the root run. Every LangChain component invoked inside it automatically becomes a child span.

Spans & attribution

Add metadata to runs so you can filter, aggregate and compare in LangSmith.

Tags Tagging runs with metadata and tags ▾
@traceable( run_type="chain", name="rag_pipeline", tags=["rag", "production"], # freeform string labels metadata={"user_id": "u_1234", # structured key-value data "session": session_id} ) def rag_pipeline(query: str) -> str: ...
tags
Use for categorical labels you filter on: environment, feature flag, user tier. Tags show up as chips in the UI run list.
metadata
Use for structured data you want to query: user_id, session_id, model version, A/B variant. Metadata is searchable via the filter bar.
Sessions Grouping runs into sessions ▾
import uuid from langsmith import traceable session_id = str(uuid.uuid4()) # one per user session @traceable( run_type="chain", metadata={"session_id": session_id} ) def chat_turn(message: str) -> str: ...
Sessions aren't a native LangSmith primitive — use session_id in metadata and filter by it. LangSmith Projects act as the top-level bucket; sessions are a filtering convention within a project.
Runtime Updating metadata at runtime ▾
from langsmith import get_current_run_tree @traceable(run_type="chain") def process(q: str) -> str: run = get_current_run_tree() run.add_metadata({"tokens_used": token_count}) run.add_tags(["flagged_for_review"]) return answer
get_current_run_tree() returns the active run so you can attach computed info (e.g. actual token counts or retrieved doc IDs) after they are known.

Prompt versioning in LangSmith

Version-control prompts in the Hub; pull pinned commits in code.

Push Publishing a prompt to the Hub ▾
from langsmith import Client from langchain_core.prompts import ChatPromptTemplate client = Client() prompt = ChatPromptTemplate.from_messages([ ("system", "You are a helpful assistant. Context: {context}"), ("human", "{question}"), ]) # Push creates a new commit; returns the commit hash commit_hash = client.push_prompt("rag-assistant", object=prompt)
Every push_prompt creates an immutable commit. The same name with a new push is a new version — old commits remain accessible forever.
Pull Pulling a prompt (latest vs pinned) ▾
from langchain import hub # Always gets the latest commit — fine for development prompt_latest = hub.pull("rag-assistant") # Pin to a commit hash for production safety prompt_pinned = hub.pull("rag-assistant:abc12345")
Always pin prompts with a commit hash in production. pull("name") follows latest, which can silently change behavior when a teammate edits the prompt.
Link Linking prompt versions to traces ▾
from langsmith import traceable from langchain import hub # Fetch prompt once at startup, tag its version in metadata prompt_obj = hub.pull("rag-assistant:abc12345") @traceable(metadata={"prompt_commit": "abc12345"}) def answer(q: str) -> str: chain = prompt_obj | llm return chain.invoke({"question": q}).content
Storing prompt_commit in run metadata lets you filter LangSmith by commit hash to compare how different prompt versions affect outputs, latency and cost side-by-side.

Debugging failures

A repeatable workflow for root-causing bad outputs or errors in your agent.

1
Find the failing trace
In LangSmith, filter by Error status or by your custom tag (flagged, user_report). Traces with exceptions show a red badge.
2
Inspect the span tree
Expand the tree to find the failing span. Check inputs, outputs and the error message at each level. LLM spans show the exact prompt sent and the token usage.
3
Attach feedback programmatically
Log structured feedback (thumbs, score, corrections) to runs so you can build evaluation datasets directly from production failures.
from langsmith import Client client = Client() client.create_feedback( run_id=failing_run_id, key="correctness", score=0, comment="Wrong retrieval source" )
4
Replay in the Playground
Click Open in Playground on any LLM span to edit the exact prompt that caused the failure and re-run it live. Iterate on the prompt without touching code.
5
Add to a dataset and run evals
Add the failing input → expected output pair to a LangSmith Dataset. Run evaluators against it after each fix to prevent regressions.
Never debug from logs alone — without the full span tree you can't see whether a failure came from the retriever, the LLM call, or a downstream tool. Always anchor debugging to a LangSmith trace.
