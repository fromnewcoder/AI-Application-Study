Agent execution loop
The agent follows a reason → act → observe loop. The LLM decides which tool to call; the tool node executes it and returns results; the LLM reasons again — repeating until it has enough information to answer.

Why LangGraph?
LangGraph's explicit graph lets you control the loop: the agent can call multiple tools in sequence, observe results mid-conversation, and loop back to reason again — without hacking around a linear chain.
ReAct pattern
This agent implements ReAct: Reason + Act. The LLM first reasons about which tool to use, then acts by invoking it, then observes the result before reasoning again. The loop runs until no tool calls are needed.
Tool binding
llm.bind_tools(TOOLS) attaches tool schemas to the LLM. When the model wants to call a tool it emits a structured tool_calls field on the AIMessage instead of plain text.
Recursion limit
Always pass recursion_limit to invoke(). Without it, a confused agent can loop indefinitely. The default in these examples is 20 — enough for complex multi-tool chains without runaway costs.
Key design decisions
Single agent node
One node for all LLM reasoning. Keeps the graph simple; add specialised nodes later if needed.
Single tool node
Executes all tool calls from one LLM turn in parallel order, then returns all results at once.
add_messages reducer
Appends new messages to history — never overwrites. This gives the LLM full context across all turns.
tool_calls_log
A separate audit list (not in messages) tracks every tool invocation with timestamps and result previews.
Safe calculator
Uses AST parsing to whitelist only known functions — no eval() on raw strings, no arbitrary code execution.
AgentState — the shared memory
Every node reads from and writes to this TypedDict. LangGraph merges partial updates automatically — nodes only return the keys they changed.
messages
Full conversation history: HumanMessage → AIMessage → ToolMessage → AIMessage → …
Uses the add_messages reducer so new messages are appended, not replaced.
tool_calls_log
Audit trail of every tool invocation. Each entry records timestamp, tool name, args, and a result preview. Separate from messages — the LLM never sees this directly.
task_label
Human-readable label for the current task. Used in logging and the summary table. Not sent to the LLM.

The add_messages reducer
Normally a LangGraph node returning {"messages": new_list} would replace the state. The Annotated[list, add_messages] type annotation changes this — LangGraph instead appends new messages, preserving the full history. This is what gives the agent memory across multiple tool calls in one session.

Tool anatomy — how @tool works
The @tool decorator from LangChain converts a plain Python function into a StructuredTool. It extracts the function's name, docstring (as the tool description), and type annotations (as the input schema). The LLM sees only the description and schema — your implementation stays hidden. This schema is what bind_tools() sends to the model so it knows exactly how to call each tool.
Safety: the calculator's AST guard
A naive eval(expression) would let the LLM execute any Python — including os.system("rm -rf /"). The calculator uses ast.parse() to walk the expression tree before evaluation and rejects any function call or name that isn't in an explicit whitelist of math functions. Only known safe names like sqrt, factorial, and pi are permitted.
1 — Define state with add_messages
from typing import Annotated, TypedDict from langgraph.graph.message import add_messages class AgentState(TypedDict): messages: Annotated[list, add_messages] # appends, not replaces tool_calls_log: list[dict] # audit trail task_label: str # for logging
2 — Declare tools with @tool
from langchain_core.tools import tool @tool def calculator(expression: str) -> str: """Safely evaluate a mathematical expression.""" import ast, math ALLOWED = {"sqrt": math.sqrt, "pi": math.pi, ...} tree = ast.parse(expression, mode="eval") # walk tree, reject unknown names... result = eval(compile(tree, "", "eval"), {"__builtins__": {}}, ALLOWED) return f"Result: {result}"
3 — Bind tools to the LLM
TOOLS = [web_search, calculator, file_read, file_write, file_list, get_weather] TOOL_MAP = {t.name: t for t in TOOLS} llm = ChatOpenAI(model="gpt-4o-mini", temperature=0) llm_with_tools = llm.bind_tools(TOOLS) # schema → model
4 — Agent node (reason)
def agent_node(state: AgentState) -> dict: msgs = [SystemMessage(content=SYSTEM_PROMPT)] + state["messages"] response = llm_with_tools.invoke(msgs) # may include tool_calls return {"messages": [response]}
5 — Tool node (act + log)
def tool_node(state: AgentState) -> dict: last_msg = state["messages"][-1] tool_messages, log = [], list(state["tool_calls_log"]) for tc in last_msg.tool_calls: result = TOOL_MAP[tc["name"]].invoke(tc["args"]) tool_messages.append(ToolMessage( content=str(result), tool_call_id=tc["id"], name=tc["name"])) log.append({"tool": tc["name"], "args": tc["args"], ...}) return {"messages": tool_messages, "tool_calls_log": log}
6 — Router + compile
def should_continue(state) -> str: last = state["messages"][-1] if hasattr(last, "tool_calls") and last.tool_calls: return "tools" # keep looping return END # final answer reached graph = StateGraph(AgentState) graph.add_node("agent", agent_node) graph.add_node("tools", tool_node) graph.set_entry_point("agent") graph.add_conditional_edges("agent", should_continue, {"tools": "tools", END: END}) graph.add_edge("tools", "agent") # loop back app = graph.compile()
