Chains
Sequential call sequences
A Chain is the original LangChain primitive — it strings together LLM calls, tools, and transformations in a fixed sequence.

Think of it like a Unix pipe: output of step N feeds input of step N+1. 
# Classic LLMChain (legacy)
from langchain.chains import LLMChain
chain = LLMChain(
    llm=llm,
    prompt=prompt_template
)
result = chain.run("What is quantum computing?")
Chains are being superseded by LCEL Runnables, but you'll see them widely in older code.
▶
Runnables
The universal interface
Every modern LangChain component implements Runnable — a unified interface with .invoke(), .stream(), .batch(), and .ainvoke().

This means prompts, LLMs, retrievers, and output parsers are ALL Runnables and can be piped together. 
# Any Runnable can be invoked the same way
result = runnable.invoke(input)
# Stream tokens as they arrive
for chunk in runnable.stream(input):
    print(chunk, end="")
# Batch multiple inputs in parallel
results = runnable.batch([input1, input2])
|
LCEL
LangChain Expression Language
LCEL uses Python's pipe operator | to compose Runnables into pipelines. It provides streaming, async, batching, and tracing out of the box — all from one composition. 
# LCEL chain with pipe operator
chain = prompt | llm | StrOutputParser()

# Invocation feels natural
result = chain.invoke({
    "topic": "quantum computing"
})

# Streaming just works
for chunk in chain.stream({"topic": "..."}):
    print(chunk, end="")
The pipe | creates a RunnableSequence under the hood.

Prompts
PromptTemplate & ChatPromptTemplate
Prompt templates are reusable, parametrised message builders. LangChain has two main kinds:

PromptTemplate — single-string prompts with {variables}
ChatPromptTemplate — list of messages (system + human + AI roles) 
from langchain_core.prompts import ChatPromptTemplate

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful {role}."),
    ("human",  "{question}"),
])

# Format to messages list
msgs = prompt.format_messages(
    role="tutor", question="Explain LCEL"
)

Output Parsers
Structured output extraction
Output parsers transform raw LLM text into structured data. Common ones:

StrOutputParser — plain string
JsonOutputParser — dict/JSON
PydanticOutputParser — typed Pydantic model
CommaSeparatedListOutputParser — list 
from langchain_core.output_parsers import JsonOutputParser
from pydantic import BaseModel

class Movie(BaseModel):
    title: str
    year: int
    genre: str

parser = JsonOutputParser(pydantic_object=Movie)
chain = prompt | llm | parser
movie = chain.invoke({"query": "Inception"})
# → Movie(title='Inception', year=2010, ...)

RunnableParallel
Parallel branches
Run multiple chains simultaneously and merge their outputs using RunnableParallel (or a plain dict {} shorthand in LCEL). 
from langchain_core.runnables import RunnableParallel

# Dict shorthand = RunnableParallel
parallel = {
    "summary": summarize_chain,
    "keywords": keywords_chain,
}

chain = parallel | combine_chain
result = chain.invoke({"text": document})
# result = {"summary": "...", "keywords": [...]}
Both branches run concurrently. LangChain handles threading automatically.

LCEL Key Properties
S
Streaming — Call .stream() on any chain and get tokens as they arrive. Zero extra code.
A
Async — .ainvoke() and .astream() work on every Runnable with no extra configuration.
B
Batching — .batch([…]) processes many inputs in parallel with automatic concurrency control.
T
Tracing — Every run is automatically traced in LangSmith when configured. No decorators needed.
F
Fallbacks — Chain .with_fallbacks([backup_llm]) retries with alternatives on failure.
Here's how to rebuild a full chatbot (with memory) using LCEL. Toggle between a plain Python version and the LCEL version to see the difference.
Chatbot with Memory
# Plain Python chatbot — explicit loop, manual history from openai import OpenAI client = OpenAI() history = [] def chat(user_msg): history.append({"role": "user", "content": user_msg}) response = client.chat.completions.create( model="gpt-4o", messages=[ {"role": "system", "content": "You are a helpful assistant."}, *history ] ) reply = response.choices[0].message.content history.append({"role": "assistant", "content": reply}) return reply # Usage while True: user = input("You: ") print("Bot:", chat(user))
# LCEL chatbot — streaming, async, tracing built-in from langchain_openai import ChatOpenAI from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder from langchain_core.output_parsers import StrOutputParser from langchain_community.chat_message_histories import ChatMessageHistory from langchain_core.runnables.history import RunnableWithMessageHistory # 1. Define prompt with history placeholder prompt = ChatPromptTemplate.from_messages([ ("system", "You are a helpful assistant."), MessagesPlaceholder("chat_history"), ("human", "{input}"), ]) # 2. Build the core LCEL chain chain = prompt | ChatOpenAI(model="gpt-4o") | StrOutputParser() # 3. Wrap with session memory — one store, many sessions store = {} def get_session_history(session_id: str): if session_id not in store: store[session_id] = ChatMessageHistory() return store[session_id] chatbot = RunnableWithMessageHistory( chain, get_session_history, input_messages_key="input", history_messages_key="chat_history", ) # 4. Stream responses with session isolation config = {"configurable": {"session_id": "user_42"}} for chunk in chatbot.stream({"input": "Hello!"}, config=config): print(chunk, end="", flush=True)
What LCEL buys you here: streaming tokens land token-by-token, multiple user sessions are isolated automatically, async support requires zero extra code, and LangSmith traces every turn with one environment variable.
When LangChain helps
Rapid prototyping — off-the-shelf chains for RAG, summarisation, Q&A
Streaming + async without boilerplate — LCEL handles it automatically
Multi-LLM swapping — change one class name to switch providers
Observability — LangSmith tracing with zero code change
Memory management — RunnableWithMessageHistory handles session state
Agent tooling — pre-built tools: web search, code exec, SQL, file I/O
Community integrations — 100+ vector stores, LLM providers, loaders
When LangChain hurts
Simple one-shot calls — direct SDK is 5 lines vs 20 with LangChain overhead
Deep debugging — abstraction layers obscure where errors originate
Tight latency budgets — extra abstraction adds milliseconds
Team unfamiliar with LCEL — steep learning curve for the | operator pattern
Unstable APIs — LangChain releases breaking changes frequently
Production fine-tuning — custom logic fights the abstraction boundaries
Dependency bloat — installs hundreds of packages even for basic use
Rule of thumb: Use LangChain when you need RAG, agents, memory, or multi-step pipelines. Skip it for single-turn LLM calls, latency-critical paths, or when you want full control over every token.

Quick Reference
Runnable
Universal interface with .invoke(), .stream(), .batch(), .ainvoke(). Everything in LCEL is a Runnable.
LCEL |
Pipe operator composes Runnables: prompt | llm | parser creates a RunnableSequence.
ChatPromptTemplate
Builds typed message lists (system, human, AI). Use MessagesPlaceholder for history injection.
StrOutputParser
Extracts .content string from an AIMessage. The simplest output parser.
JsonOutputParser
Parses LLM output to dict or Pydantic model. Adds format instructions to the prompt automatically.
RunnableParallel
Run multiple chains in parallel. Dict shorthand: {"a": chain_a, "b": chain_b} | merger.
RunnableLambda
Wrap any Python function as a Runnable: RunnableLambda(my_fn).
RunnableWithMessageHistory
Adds session-scoped chat memory to any LCEL chain. Accepts a session_id in config.
.with_fallbacks()
Retry with backup Runnables on failure: primary_llm.with_fallbacks([backup_llm]).
.with_retry()
Automatic retry with exponential backoff: llm.with_retry(stop_after_attempt=3).
MessagesPlaceholder
Slot in a ChatPromptTemplate where a list of messages (e.g., chat history) is injected at runtime.
